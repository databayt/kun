# CLAUDE.md

The most powerful configuration primitive in Claude Code. A markdown file that shapes every response, every agent, every workflow. Kun uses a three-layer hierarchy that cascades from global preferences down to repo-specific operations.

## Three-Layer Hierarchy

```
Priority (high → low):
1. Project-level    <project>/CLAUDE.md         — Project context
2. Repo-level       <project>/.claude/CLAUDE.md — Keywords, workflows
3. User-level       ~/.claude/CLAUDE.md         — Stack, preferences
4. Pattern library  codebase repo               — Core patterns
```

Higher layers override lower ones — project-specific context wins over global defaults.

## Layer 1: User-level (`~/.claude/CLAUDE.md`)

Global preferences that apply to every project and conversation.

| Setting | Value |
|---------|-------|
| Model | `claude-opus-4-7` |
| Package manager | pnpm |
| Stack | Next.js 16, React 19, Prisma 6, TypeScript 5, Tailwind CSS 4, shadcn/ui |
| Languages | Arabic (RTL default), English (LTR) |
| Port | Always 3000 |
| Environment | Single `.env` only |
| Component hierarchy | ui, atom, template, block, micro |

## Layer 2: Project-level (`CLAUDE.md`)

Project-specific context: domain model, multi-tenancy approach, license, key constraints.

## Layer 3: Repo-level (`.claude/CLAUDE.md`)

Operational configuration: keyword-to-action mappings, MCP trigger table, slash command reference, organization repo references, framework best practices.

### What It Contains

- **Keyword mappings** — `dev` → start server, `push` → commit, `deploy` → ship
- **Component hierarchy** — ui, atom, template, block, micro
- **MCP trigger table** — which keyword activates which server
- **Slash command reference** — every available `/` command
- **Organization references** — how to reference other repos
- **Framework best practices** — performance, RTL, RSC patterns

## Writing Effective CLAUDE.md

### Do

- Lead with preferences (model, stack, language)
- Map keywords to actions (`say X → do Y`)
- Define the component hierarchy
- Reference external repos and patterns
- Include framework best practices

### Avoid

- Implementation details (that's what agents and rules are for)
- Long prose (tables and lists are clearer)
- Duplicating what's in rules or agents
- Sensitive information (use secrets instead)
