# Issue

The issue pipeline connects user-reported bugs to automated fixes. Users click "Report an Issue" in the UI, a GitHub issue is created, and the report agent processes it.

## Report an Issue (User-Facing)

Every product page has a "Report an Issue" link or icon. When clicked, a dialog opens with a textarea. On submit, a GitHub issue is created in the relevant repo with the `report` label.

### Where It Appears

| Location | Variant |
|----------|---------|
| SaaS marketing footer | `text` |
| Application form wizard | `icon` |
| School marketing pages | `text` |
| School dashboard | `text` |
| SaaS dashboard | `text` |
| Auth pages | `text` |
| Onboarding | `text` |

### Component

`src/components/report-issue.tsx`

Two variants:
- **`text`** (default) ŌĆö underlined "Report an issue" link
- **`icon`** ŌĆö bug icon (`lucide-react`)

Bilingual via `useDictionary()`.

### Server Action

`src/lib/actions/report-issue.ts`

On submit:
1. Reads `GITHUB_PERSONAL_ACCESS_TOKEN` from env
2. Reads auth session via `auth()`
3. Creates GitHub issue with `report` label
4. Posts an acknowledgment comment

### Issue Body Format

```markdown
<user's description>

---

**Page**: <product-url>
**Time**: <ISO timestamp>
```

Title is truncated to 80 characters from the description.

## Report Agent (Auto-Fix)

The report agent processes open `report`-labeled issues. Triggered manually by `report` in Claude Code or by the session-start hook.

### Trigger

```bash
report
report <product>
```

Session-start hook checks repos for open report issues automatically.

### Pipeline

```
READ ŌåÆ LOCATE ŌåÆ CONTEXT ŌåÆ VALIDATE ŌåÆ FIX ŌåÆ BUILD ŌåÆ PUSH ŌåÆ CLOSE
```

| Step | What It Does |
|------|--------------|
| **READ** | `gh issue view` ŌĆö extract description, page URL, time |
| **LOCATE** | Map URL to route + component directories |
| **CONTEXT** | Read `CLAUDE.md`, `README.md`, `ISSUE.md` |
| **VALIDATE** | Real bug? Aligned? Will the fix improve things? |
| **FIX** | Edit minimum code, follow documented patterns |
| **BUILD** | `pnpm build` ŌĆö verify no regressions |
| **PUSH** | `fix:` commit with `Closes #N` |
| **CLOSE** | Close with structured comment |

### Validation Gate

1. **Real bug?** ŌĆö can it reproduce? If not ŌåÆ `cannot-reproduce`
2. **Aligned?** ŌĆö feature request? ŌåÆ `needs-human`
3. **Safe fix?** ŌĆö could it break shared code? ŌåÆ `needs-human`

### Escalation

| Situation | Action |
|-----------|--------|
| Build error after fix | Hand off to `build` agent |
| Server-side exception | Hand off to `sse` agent |
| Performance issue | Hand off to `performance` agent |
| Security concern | Hand off to `guardian` agent |
| Architecture change | `needs-human` label |
| Cannot reproduce | `cannot-reproduce` label |

## Scheduled Trigger

A remote trigger can run the report agent on a schedule in Anthropic's cloud ŌĆö typically every 2 hours.

| Field | Value |
|-------|-------|
| **Schedule** | `0 */2 * * *` (every 2 hours) |
| **Model** | `claude-sonnet-4-6` |
| **Manage** | https://claude.ai/code/scheduled |

## Full Pipeline Flow

```
User clicks "Report an issue"
    Ōåō
Dialog opens ŌåÆ user types description ŌåÆ Submit
    Ōåō
Server action creates GitHub issue (label: report)
    Ōåō
Acknowledgment comment posted
    Ōåō
Next Claude Code session OR /report keyword OR scheduled trigger
    Ōåō
Agent: READ ŌåÆ LOCATE ŌåÆ CONTEXT ŌåÆ VALIDATE ŌåÆ FIX ŌåÆ BUILD ŌåÆ PUSH
    Ōåō
Issue closed with structured comment
    Ōåō
Vercel auto-deploys fix to production
```

## Issue Creation (CLI)

```bash

```

### Labels

| Label | When |
|-------|------|
| `P0-critical` | Production down, data loss, security |
| `P1-high` | Blocks current sprint |
| `P2-medium` | Important but not blocking |
| `P3-low` | Nice to have, cleanup |
| `report` | User-reported via dialog |
| `bug` | Something is broken |
| `enhancement` | Improve existing feature |
| `needs-human` | Agent can't fix |
| `cannot-reproduce` | Agent couldn't reproduce |
